# RuView — 基于 WiFi 信号的隐私优先人体感知系统

> **项目地址**: [https://github.com/ruvnet/RuView](https://github.com/ruvnet/RuView)
> **许可证**: MIT | **语言**: Rust (主) + Python | **Stars**: 43.8k+

---

## 一、项目简介

RuView 是一个**边缘 AI 感知系统**，利用 WiFi 信号（而非摄像头）实现人体姿态估计、生命体征监测和存在检测。其核心原理是：WiFi 路由器不断向空间发射无线电波，当人体移动甚至呼吸时，这些电波的散射模式会发生变化。RuView 通过分析 WiFi 的**信道状态信息（CSI）**捕捉这些变化，从而在不采集任何视觉数据的前提下感知人体活动。

该项目源于卡内基梅隆大学的学术论文 *DensePose From WiFi*（arXiv:2301.00250），并将其扩展为可在低成本硬件上实际部署的边缘系统。

## 二、核心能力

| 能力 | 实现方式 | 性能指标 |
|------|----------|----------|
| **人体姿态估计** | CSI 子载波幅度/相位 → DensePose UV 映射 | 54,000 fps（Rust） |
| **呼吸检测** | 带通滤波 0.1-0.5 Hz → FFT 峰值提取 | 6-30 BPM |
| **心率检测** | 带通滤波 0.8-2.0 Hz → FFT 峰值提取 | 40-120 BPM |
| **存在感知** | RSSI 方差 + 运动频带功率 | < 1ms 延迟 |
| **穿墙感知** | 菲涅尔区几何 + 多径建模 | 最远 5 米 |

## 三、技术架构

### 3.1 端到端信号处理流水线

```
WiFi 路由器发射无线电波
    ↓
ESP32 网格（4-6 节点）通过 TDM 协议在信道 1/6/11 捕获 CSI
    ↓
多频段融合: 3 信道 × 56 子载波 = 168 虚拟子载波/链路
    ↓
多站点融合: N×(N-1) 链路 → 注意力加权跨视角嵌入
    ↓
相干门控: 接受/仅预测/拒绝 → 无需调参，数日稳定
    ↓
信号处理: Hampel 滤波、SpotFi 相位校正、菲涅尔模型、BVP、频谱图
    ↓
AI 骨干（RuVector）: 注意力机制、图算法、压缩、场模型
    ↓
神经网络: 处理后的信号 → 17 个身体关键点 + 生命体征 + 房间模型
    ↓
输出: 实时姿态、呼吸、心率、房间指纹、漂移告警
```

### 3.2 关键信号处理算法（6 项学术级算法）

| 算法 | 功能 | 学术来源 |
|------|------|----------|
| **共轭乘法** | 消除 CFO/SFO 引起的相位污染 | SpotFi (SIGCOMM 2015) |
| **Hampel 滤波器** | 基于中位数的异常值剔除，抗 50% 数据污染 | 标准 DSP |
| **菲涅尔区模型** | 基于 TX-RX-人体几何建模呼吸引起的信号变化 | FarSense (MobiCom 2019) |
| **CSI 频谱图** | 滑动窗口 FFT，生成 2D 时频矩阵 | 2018 年后标准方法 |
| **子载波选择** | 按运动敏感度排序，选取 Top-K 子载波 | WiDance (MobiCom 2017) |
| **体速度剖面（BVP）** | 从多普勒频移提取速度分布，实现跨环境识别 | Widar 3.0 (MobiSys 2019) |

### 3.3 AI 骨干：RuVector

RuVector 是智能信号处理层，用学习型算法替代手工阈值：

| AI 能力 | 替代的传统方法 | 效果 |
|---------|---------------|------|
| 自优化信道选择 | 固定阈值（换房间就失效） | 图最小割自动适应任何环境 |
| 注意力信号清洗 | 固定能量截断（遗漏微弱呼吸） | 学习型门控放大人体信号 |
| 学习型信号融合 | 简单平均（一个坏信道污染全部） | Transformer 注意力降权损坏信道 |
| 物理约束定位 | 昂贵的非线性求解器 | 稀疏最小二乘菲涅尔几何实时求解 |
| 75% 内存压缩 | 13.4 MB 呼吸缓冲区（边缘设备溢出） | 分层 3-8 bit 量化，60s 生命数据仅 3.4 MB |

## 六、自学习 WiFi AI（ADR-024）

系统能从原始 WiFi 信号自主学习，无需摄像头或标注数据：

- 将 WiFi 信号转化为 128 维"指纹"向量
- 自动识别房间、检测入侵者、分类活动
- **模型极小**：~55,000 参数 / 55 KB 内存，可跑在 $8 的 ESP32 上
- 支持 MicroLoRA 适配器（每个房间仅 1,792 参数），迁移到新房间仅需原训练数据的 7%

```
WiFi 信号 [56 信道] → Transformer + 图神经网络
                           ├→ 128 维环境指纹（搜索 + 识别）
                           └→ 17 关节人体姿态（人体追踪）
```

## 七、跨环境泛化（Project MERIDIAN）

解决 WiFi 感知的最大痛点——"换个房间模型就失效"：

| 技术 | 原理 | 效果 |
|------|------|------|
| 梯度反转层 | 对抗训练迫使模型丢弃房间特定的捷径 | 环境无关的特征 |
| 几何编码器（FiLM） | 傅里叶编码收发器位置作为逐层调制 | 模型知道硬件位置 |
| 硬件归一化 | 将任何芯片的 CSI 重采样为标准 56 子载波格式 | Intel 5300 和 ESP32 数据对齐 |
| 虚拟域增强 | 随机生成合成房间参数 | 仅 2-3 个房间数据训练可见 1000+ |
| 测试时训练（TTT） | 少量未标注帧即可自适应 | 零样本部署 |

## 十、快速体验

### Docker 一键启动（无需任何工具链）

```bash
docker pull ruvnet/wifi-densepose:latest
docker run -p 3000:3000 -p 3001:3001 -p 5005:5005/udp ruvnet/wifi-densepose:latest
# 打开 http://localhost:3000
```

### 从源码构建

```bash
git clone https://github.com/ruvnet/RuView.git
cd RuView

# Rust（推荐）
cd rust-port/wifi-densepose-rs
cargo build --release
cargo test --workspace

# Python（遗留版）
pip install -r requirements.txt
pip install -e .
```

### REST API

```bash
curl http://localhost:3000/health              # 健康检查
curl http://localhost:3000/api/v1/sensing/latest  # 最新感知帧
curl http://localhost:3000/api/v1/vital-signs     # 生命体征
curl http://localhost:3000/api/v1/pose/current    # 姿态估计
```

### WebSocket 实时流

```python
import asyncio, websockets, json

async def stream():
    async with websockets.connect("ws://localhost:3001/ws/sensing") as ws:
        async for msg in ws:
            data = json.loads(msg)
            print(f"检测到 {len(data.get('persons', []))} 人")

asyncio.run(stream())
```

## 十一、应用场景一览

### 日常场景（普通 WiFi 即可）

- **养老护理** — 跌倒检测、夜间活动监测、睡眠呼吸率（无需佩戴设备）
- **医院病房** — 非关键床位连续呼吸 + 心率监测
- **零售门店** — 客流量、停留热图、排队长度（GDPR 友好）
- **办公空间** — 实际工位使用率、会议室空置检测、HVAC 优化

### 极端场景（需 CSI 硬件）

- **消防搜救** — 烟雾和墙壁后方定位被困人员
- **军事/战术** — 穿墙人员检测、房间清理确认
- **矿井/地下** — GPS/摄像头失效时追踪人员
- **协作机器人** — 检测遮挡物后方的人员，自动减速或停止

## 十二、项目状态与注意事项

> ⚠️ **Alpha 阶段** — 仍在活跃开发中

**已知限制：**
- 多节点人数计数可能不够准确（#249）
- 训练精度可能停滞在较低 PCK 值（#318），超参数调优进行中
- **未提供预训练权重**，需从头训练
- ESP32-C3 和初代 ESP32 不支持（单核，CSI DSP 能力不足）
- 单个 ESP32 部署空间分辨率有限

**项目规模：**
- 325 次提交 / 7 位贡献者
- 62 个架构决策记录（ADR）
